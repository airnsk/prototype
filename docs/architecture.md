# Архитектурное описание прототипа Direct-ZTNA

> Детальное описание архитектуры, моделей данных, протоколов взаимодействия и механизмов безопасности.

---

## 1. Архитектурные принципы

### 1.1. Разделение плоскостей управления и данных

Direct-ZTNA строго разделяет две плоскости:

- **Control Plane** — принятие решений (Policy Decision Point, PDP).
  - Аутентификация, авторизация, выпуск разрешений, отзыв доступа.
  - Централизован, но не обрабатывает пользовательский трафик.

- **Data Plane** — передача прикладных данных.
  - Только прямые соединения между конечными агентами.
  - Нет постоянного транзитного посредника (шлюза, прокси, VPN-концентратора).

### 1.2. По-сеансовый доступ

Доступ к сервису не является статическим свойством пользователя или устройства. Он существует только как временное состояние системы:

1. Создаётся контроллером (выдача билета).
2. Применяется агентами (активация TUN-peer).
3. Удаляется при истечении срока или принудительном отзыве.

### 1.3. Два фактора доступа

Открытие канала требует одновременного выполнения двух независимых условий:

1. **Билет** (`AccessTicket`) — подписанное разрешение контроллера, содержащее параметры сессии.
2. **PSK** — предварительно распределённый симметричный ключ, привязанный к паре «клиент–сервис».

Это реализует принцип **separation of duties**: компрометация одного фактора недостаточна для доступа.

## 3. Модели данных

### 3.1. AccessTicket (криптографическое разрешение)

```python
@dataclass
class AccessTicket:
    iss: str              # "ztna-controller"
    aud: str              # "client" или "server"
    sub: str              # идентификатор пользователя
    service_id: str       # целевой сервис
    jti: str              # уникальный ID билета
    nbf: int              # not before (unix timestamp)
    iat: int              # issued at (unix timestamp)
    exp: int              # expiration (unix timestamp)
    key_id: str           # ссылка на PSK
    transport: TransportProfile
    scope: str            # "tcp:443" или подобное
    sig: str              # Ed25519 подпись (hex)
```

**Поле `jti` (JWT ID):**
- Уникален для каждого билета.
- Используется для отзыва: контроллер добавляет `jti` в множество `Revoked`.
- Защита от replay: даже если билет перехвачен, повторное использование возможно только до истечения срока и только при отсутствии в `Revoked`.

**Поле `transport`:**
```python
@dataclass
class TransportProfile:
    server_pubkey: str    # публичный ключ сервера TUN-шифратора
    endpoint: str         # IP:port (например, "172.21.0.50:51820")
    allowed_ips: List[str] # CIDR-список разрешённых адресов
```

### 3.2. Каноническая сериализация для подписи

Для обеспечения детерминированности подписи payload сериализуется следующим образом:

```python
json.dumps(payload, sort_keys=True, separators=(",", ":"), ensure_ascii=False)
```

Это гарантирует, что одинаковый payload всегда даёт одинаковые байты для подписи, независимо от порядка ключей в словаре.

### 3.3. RevokeCommand

```python
@dataclass
class RevokeCommand:
    jti: str
    revoked_at: int       # unix timestamp
    reason: Optional[str]
```

### 3.4. MetricEvent (телеметрия)

```python
@dataclass
class MetricEvent:
    node: str             # controller, client-agent, server-agent, ...
    event: str            # request, auth_done, policy_done, ticket_issued,
                          # peer_add, tunnel_up, first_data, revoke_cmd,
                          # peer_remove, traffic_stop
    ts: float             # time.time() с точностью до микросекунд
    ticket_id: Optional[str]
    details: Optional[dict]
```

## 4. Криптографическая модель

### 4.1. Ed25519 для подписи билетов

- **Генерация:** `Ed25519PrivateKey.generate()` (32 байта seed).
- **Сохранение:** raw bytes (32 байта для приватного, 32 байта для публичного).
- **Подпись:** 64-байтовая Ed25519 подпись, сериализуется в hex.

### 4.2. Предварительно распределённые ключи (PSK)

- Формат: 256-битный ключ, base64-кодированный.
- Хранение: JSON-файлы (`client_keys.json`, `server_keys.json`).
- Связность: один и тот же PSK для пары «клиент–сервис» загружается в оба агента.
- В промышленной реализации должен заменяться на KMS/HSM/TPM.

### 4.3. TUN-шифратор как транспорт

- **Роль:** обеспечивает прямое защищённое соединение и «криптографическое молчание» (silent drop).
- **Криптография:** AES-256-GCM через библиотеку `cryptography` (Python), аутентифицированное шифрование с nonce (12 байт) и tag (16 байт).
- **PSK:** предварительно распределённый симметричный ключ, привязанный к паре «клиент–сервис».
- **Silent drop:** пакеты без валидного ключа отбрасываются без ответа. Сервис не виден в сети до выдачи билета.

## 5. Policy Engine

### 5.1. Текущая реализация (прототип)

Rule-based проверка двух условий:

1. Сервис существует в реестре (`registry.json`).
2. Сервис находится в `allowed_services` клиентского агента.

### 5.2. Пример реестра

```json
{
  "services": {
    "protected-service-1": {
      "endpoint": "172.21.0.50:51820",
      "server_pubkey": "queZz9wjmg8VPbKe1/IU+6Tg+sKcIQTuXBVdTATEK48=",
      "allowed_ips": ["172.21.0.50/32"]
    }
  },
  "agents": {
    "client-agent": {
      "allowed_services": ["protected-service-1"]
    }
  }
}
```

### 5.3. Расширяемость

В промышленной реализации Policy Engine может включать:
- RBAC/ABAC (роли, группы, атрибуты).
- Временные политики (рабочие часы, дни недели).
- Геолокацию и контекст устройства.
- Непрерывную оценку риска.

## 6. Жизненный цикл сессии

### 6.1. Стадии

| Стадия | Действие | Ответственный |
|--------|----------|---------------|
| **Establish** | Запрос → Auth → Policy → Ticket → Notify server → Local validation → TUN up | Контроллер, агенты |
| **Active** | Прямая передача прикладного трафика | Клиентский и серверный агенты |
| **Expire** | Автономное удаление peer по `exp` | Серверный агент (SessionMonitor) |
| **Revoke** | Команда отзыва → удаление peer → cleanup | Контроллер, агенты |

### 6.2. Граничные случаи

**Потеря связи с контроллером при revoke:**
- Если команда `RevokeCommand` не доходит до агента, доступ прекратится по истечении `exp` (TTL).
- Для критичных сценариев рекомендуется короткий TTL + heartbeat.

**Оффлайн-клиент:**
- Серверная сторона гарантированно закрывает доступ при получении revoke.
- Клиент узнаёт об отзыве только при восстановлении связи.

**Массовый revoke:**
- Batch-обработка и приоритизация команд.
- Локальное кэширование списка отозванных jti.

## 7. Модель угроз и контрмеры

| Угроза | Компонент | Мера снижения риска |
|--------|-----------|---------------------|
| Подделка билета | Контроллер | Подпись Ed25519, компрометация ключа требует доступа к `controller.sk` |
| Replay билета | Сессия | Уникальный `jti` + множество `Revoked` + TTL |
| Компрометация PSK | Агент | Soft-token (в прототипе); TPM/HSM в продакшене |
| Компрометация endpoint | Клиент | Доступ только к сервисам с действующим билетом и PSK |
| Компрометация контроллера | Control Plane | Изоляция, HSM для ключа подписи, аудит |
| MITM при установлении | Транспорт | TUN-шифратор с PSK, привязка к конкретным endpoint |
| Горизонтальное перемещение | Сеть | Изоляция по-сеансовая, нет сетевой видимости без билета |

## 8. Ограничения прототипа

1. **Нет PKI:** открытый ключ контроллера распространяется вручную при развёртывании.
2. **Нет KMS:** жизненный цикл PSK (генерация, ротация, отзыв) не автоматизирован.
3. **Нет HSM/TPM:** ключи хранятся в файловой системе.
4. **Mock IdP:** нет MFA, федерации, непрерывной оценки контекста.
5. **L3/L4 контроль:** нет L7-инспекции (HTTP paths, methods).
6. **Нет NAT traversal:** для клиентов за symmetric NAT потребуется STUN/TURN/ретранслятор.
7. **Один контроллер:** нет резервирования или консенсуса.

## 9. Сравнение с формальной моделью

### 9.1. Поверхность атаки

| Архитектура | Формула | Значение в прототипе |
|-------------|---------|----------------------|
| VPN | \|U\|·\|S\| | 1·1 = 1 (в стенде) |
| Gateway-ZTNA | \|U\| + Σ\|S(g)\| | 1 + 1 = 2 |
| **Direct-ZTNA** | **K_avg** | **1 (активная сессия)** |

В прототипе `K_avg` измеряется телеметрией как число одновременно активных сессий.

### 9.2. Задержки

| Компонент | Direct-ZTNA | Примечание |
|-----------|-------------|------------|
| T_setup | ~300 ms | Auth + Policy + Sign + TUN setup |
| T_revoke | ~200 ms | Удаление peer локально, без задержки шлюза |
| L_data | Прямой путь | Без транзитного узла |

## 10. Расширение архитектуры

### 10.1. Промышленная реализация

| Компонент | Прототип | Промышленная версия |
|-----------|----------|---------------------|
| IdP | Mock (JSON) | Keycloak / Azure AD / LDAP |
| Policy Engine | Rule-based | OPA / Cedar /自定义 движок |
| Key Store | JSON files | HashiCorp Vault / AWS KMS / PKCS#11 |
| Transport | TUN-шифратор (AES-256-GCM) | WireGuard / IPSec / собственный протокол |
| Telemetry | In-memory | Prometheus / Grafana / ELK |
| Контроллер | Один instance | Kubernetes / Consul / Raft |

### 10.2. Масштабирование

- **Control Plane:** масштабируется горизонтально через stateless контроллеры + shared Redis для `Revoked`.
- **Data Plane:** масштабируется автоматически, так как трафик не проходит через центральный узел.
